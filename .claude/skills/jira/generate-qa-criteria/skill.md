# Generate QA Acceptance Criteria

Analyzes staged or branch-level code changes alongside Jira ticket context and generates a structured QA acceptance criteria comment posted directly to the ticket.

## Prerequisites

- Atlassian MCP server must be configured
- On a feature branch with changes (not main/master)
- Branch name should contain the Jira issue key (e.g., `XYZ-123-create-qa-criteria-skill`)
- The cloudId for this project is: ``

## Workflow

### 1. Determine Changes to Analyze

Gather the code diff using this priority order:

1. Check for staged changes first:
   ```bash
   git diff --staged
   ```
2. If no staged changes, use the full branch diff against master:
   ```bash
   git diff master...HEAD
   ```
3. If neither produces output, inform the user there are no changes to analyze and stop.

Also collect the list of changed files for structural analysis:
```bash
git diff --staged --name-status
# or if using branch diff:
git diff master...HEAD --name-status
```

If the diff is very large (more than 50 files changed), use `git diff master...HEAD --stat` for a summary and focus on reading the most significant files. Prioritize:
- Entity/model files (database changes)
- Service files (business logic)
- Controller files (API endpoints)
- Cron-decorated methods (scheduled tasks)
- Module files (dependency changes)

### 2. Identify the Jira Ticket

- Parse the Jira issue key from the current branch name:
  ```bash
  git branch --show-current
  ```
  - Branch pattern: `{ISSUE-KEY}-description` (e.g., `XYZ-123-create-qa-criteria-skill`)
  - The issue key is the prefix matching the pattern `[A-Z]+-[0-9]+`
- If an argument was provided (e.g., `/generate-qa-criteria XYZ-123`), use that instead
- If no issue key is found from either source, ask the user

### 3. Fetch Jira Ticket Context

Use `mcp__atlassian__getJiraIssue` with the cloudId from Prerequisites and the extracted issue key.

Extract from the response:
- **Summary** (title)
- **Description** (contains requirements, dev notes, acceptance criteria)
- **Issue type** (Story, Task, Bug — this informs the "Type of Implementation")

If the ticket fetch fails, warn the user but continue with code-only analysis. The Jira context enriches the output but is not strictly required.

### 4. Analyze the Code Changes

Read through the diff and changed files to classify modifications into these categories. Only include categories where actual changes exist:

**Entity/Database Changes:**
- New or modified TypeORM entities
- New columns, tables, or relationships
- Migration files

**Service/Business Logic Changes:**
- New or modified service methods
- Changes to campaign orchestration, bid logic, budget rules
- New helper/utility functions

**API/Controller Changes:**
- New or modified endpoints
- Changes to request/response DTOs
- Guard or middleware changes

**Cron/Scheduled Task Changes:**
- New cron jobs or modified schedules
- Changes to feature flags controlling crons
- Modified cron logic

**Module/Configuration Changes:**
- New module registrations
- New environment variables or feature flags
- Changes to database connections

**External Integration Changes:**
- DSP API call modifications
- Pub/Sub message handling changes
- Third-party service interactions (Teams webhooks, AI category service, etc.)

### 5. Generate the QA Comment

Build the comment using the structure below. **Only include sections that are relevant to the changes.** If there are no database changes, omit "Backend Testing" entirely. Do not fabricate criteria for areas that were not changed.

```markdown
## QA Context
> Auto-generated by Claude Code - Please review and adjust as needed

### Type of Implementation
[Derive from analysis: "New feature", "Update to existing logic", "Bug fix", "Configuration change", etc. Be specific — e.g., "New feature: automated DSP campaign creation for RichAds"]

### High-Level Overview
[2-4 sentences explaining what was implemented, the data workflow, and how components interact. Write for a QA audience — focus on observable behavior, not implementation details.]

### Impacted Components/Platforms/Modules
- Internal: [list affected NestJS modules, services, entities]
- External dependencies: [list any external systems touched, or "None" if purely internal]

### Acceptance Criteria
[Group criteria by functional area. Number them continuously across all groups. Each criterion should describe an observable, testable outcome.]

**[Category Name]**
1. [Criterion]
2. [Criterion]

**[Category Name]**
3. [Criterion — continuing the numbering]
4. [Criterion]

[If a criterion cannot be verified through the UI alone, append: "(Requires developer assistance to verify)"]

### Backend Testing
[ONLY include if there are database/entity changes]

| Table | What to Validate | When to Validate | Reference Query |
|-------|-----------------|-----------------|-----------------|
| [table_name] | [what to check] | [timing] | [SQL query] |

### Test Scenarios

**Functional:**
- [Scenario that exercises the primary happy path]
- [Additional functional scenarios]

**Boundary/Edge Cases:**
[ONLY include if relevant]
- [Scenario]

**Negative:**
[ONLY include if relevant]
- [Scenario]

### Notes for QA
[ONLY include if items genuinely need special attention]
- [Items requiring developer assistance for setup or verification]
- [Special environment configuration needed]
- [Useful SQL queries for monitoring]
```

### 6. Present for Review

Display the full generated comment to the user and ask whether they want to modify anything before posting. Wait for explicit approval before proceeding.

If the user requests changes, revise and present again.

### 7. Post to Jira

After user approval, use `mcp__atlassian__addCommentToJiraIssue` with the cloudId from Prerequisites, the ticket key, and the generated markdown comment.

Confirm success by displaying the issue key and noting that the comment was posted.

## Arguments

- No arguments: Auto-detect issue key from branch name, use staged or branch diff
- `ISSUE-KEY`: Explicitly specify the Jira issue key (e.g., `XYZ-123`)

## Example Invocation

User: `/generate-qa-criteria`

Claude should:
1. Detect `XYZ-123` from branch `XYZ-123-create-qa-acceptance-criteria-generator-skill`
2. Run `git diff --staged` — if empty, fall back to `git diff master...HEAD`
3. Fetch XYZ-123 from Jira for context (summary, description, type)
4. Analyze the diff and classify changes by category
5. Generate the QA comment with only relevant sections
6. Present comment for review
7. After approval, post to XYZ-123

## Notes

- This skill does NOT modify code — it only reads changes and posts a Jira comment
- Always wait for user approval before posting to Jira
- Omit sections that are not relevant rather than filling them with "N/A" or placeholder text
- When the diff is ambiguous about user-facing behavior, ask the user for clarification rather than guessing
- Criteria should be written for a QA audience: focus on what to test and how to verify, not on code structure
- If the ticket description already has acceptance criteria, use them as a starting point and augment with implementation-specific testing guidance
- The numbering across acceptance criteria categories must be continuous (1, 2, 3... not resetting per category)
- Reference queries in the Backend Testing table should use realistic column and table names from the actual entities in the diff
